# Учет в ОСББ/ЖК

## Назначение

Приложение предназначено для бухгалтерского и абонентского учета по домам ОСББ/ЖК.
Это клиентское веб-приложение на `HTML/CSS/Vanilla JS` с авторизацией через Supabase
и доступом к данным и документам конкретных домов.

Рядом с ним находятся:

- `index.html` — основное приложение.
- `info.html` — отдельная промо-страница.
- `files/` — файловый архив по домам, годам и месяцам.

## Бизнес-правила

- Основная сущность учета: `billing_accounts`.
- Базовое правило распределения оплат: FIFO.
- Назначение платежа не считается авторитетным источником распределения.
- Для начислений, льгот по начислениям и ручных корректировок используются `posting_month` и `billing_month`.
- Для оплат используются `payment_date` и `payment_month`.
- Для распределений оплат используются `billing_month_from` и `billing_month_to`.
- Месяц хранится как целое число: `year * 12 + month`.

## Основные сценарии

### Авторизация

- Вход и регистрация идут через Supabase Auth.
- При открытии `index.html` приложение сначала проверяет сессию.
- Если сессия активна, пользователь сразу переводится в режим приложения.
- Если сессии нет, показывается форма входа и регистрации.

### Работа с домами

- Пользователь видит только дома, к которым у него есть связь через `user_homes`.
- После загрузки домов строится левое меню действий.
- Если в URL есть `homeCode` и `actionCode`, открывается соответствующий раздел.
- Иначе открывается первый доступный дом и первое действие.

### Основные разделы интерфейса

- лицевые счета
- перечень и табличные режимы
- платежи
- банк
- документы
- информация о доме
- схема дома

## Данные и доступ

### Таблицы, которые реально использует клиент

По текущему коду клиент напрямую работает как минимум с:

- `homes`
- `user_homes`
- `corrections`

### Что хранится в `homes`

Из клиентского кода видно, что из таблицы читается поле `data`, внутри которого лежит JSON
с описанием дома, действиями, файлами и справочными данными. После чтения этот JSON
парсится на клиенте.

### Файловый архив

Папка `files/` организована по иерархии:

- дом
- год
- месяц
- тип документа или подпапка

Эта структура используется модулем просмотра документов.

### Таблицы `public` с включенным RLS

- `corrections`
- `homes`
- `user_homes`

### Фактические политики

#### `homes`

- `SELECT`: пользователь видит дом, если существует строка в `user_homes`,
  где `user_homes.home_code = homes.code` и `user_homes.user_id = auth.uid()`.
- `INSERT`: разрешен одному конкретному admin user.
- `UPDATE`: разрешен одному конкретному admin user.

#### `user_homes`

- `SELECT`: только строки, где `user_id = auth.uid()`.

#### `corrections`

- есть permissive policy для `public` на `SELECT`
- есть permissive policy для `public` на `INSERT`
- есть permissive policy для `public` на `UPDATE`
- есть permissive policy для `public` на `DELETE`
- есть permissive policy для `public` на `ALL`

Практический результат проверки через `anon`-ключ:

- `homes` без авторизации возвращает пустой набор
- `user_homes` без авторизации возвращает пустой набор
- `corrections` без авторизации читается успешно

По решению владельца проекта `corrections` пока остается открытой,
поскольку в этой таблице нет секретных или критичных данных.

## Архитектурные изменения

- `index.html` больше не содержит inline-инициализацию Supabase и auth-bootstrap.
- Инициализация клиента вынесена в `js/supabase-client.js`.
- Проверка сессии, вход, регистрация и переключение режимов вынесены в `js/app-auth.js`.
- В `js/menu.js` убран повторный автозапуск `initApp()`, чтобы приложение не инициализировалось дважды.

## Legacy-очистка

- Из `js/ut.js` убран Babel/regenerator helper-пролог.
- В `js/ut.js` и `js/addstuf.js` заменены `concat`-цепочки на шаблонные строки.
- Логика работы не менялась, правки ограничены структурой кода и текстовыми строками интерфейса.

## Технические риски

- Проект работает без bundler и без модульной системы.
- Значительная часть логики находится в глобальной области видимости.
- Есть сильная зависимость от CDN-скриптов и глобальных переменных.
- В проекте остаются крупные legacy-файлы, которые стоит чистить дальше только небольшими шагами.

## Ближайшие безопасные шаги

1. Дальше выносить inline-логику из больших файлов только маленькими шагами.
2. Продолжить чистку `addstuf.js`, `table.js` и соседних модулей без изменения бизнес-правил.
3. Постепенно разделять доступ к данным, преобразование данных и DOM-рендеринг.
